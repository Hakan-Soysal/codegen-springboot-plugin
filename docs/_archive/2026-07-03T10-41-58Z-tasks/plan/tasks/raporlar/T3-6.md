## Yapılanlar (SpringEmitter.java)

Modül döngüsü yeniden sıralandı: `{Module}Wiring` dosyası artık döngü **sonunda** yazılıyor (bean
içerikleri önce biriktiriliyor), çünkü Wiring op-slice bean kayıtlarına bağımlı.

### 5.1 Request record — `requestRecordJava`
`{Op}Command.java` (isCommand) / `{Op}Query.java`: `signature.params` → `Naming.javaType` bileşenleri;
`// visibility: {visibility}` yorumu; `note` varsa record-üstü Javadoc + `realized("note", opId)`.
`realized("operation", opId)` + `realized("visibility", opId)` burada, HandlerBase ile birlikte
**bir kez**. Cross-module/cross-package custom tip (entity/type/event) parametreleri için otomatik
import çözümü (`customTypeImport`) — op-slice paketi (`app.{module}.{op}`) her zaman modül-kök
paketinden (`app.{module}`) farklı olduğundan gerekli (derlenebilirlik şartı).

### 5.2 HandlerBase — `handlerBaseJava`
`{Op}HandlerBase.java` (abstract, gen-owned, WriteAlways): `access.reads∪creates∪updates∪deletes`
entity'lerinin ordinal-distinct listesi (`accessEntitiesOrdinalDistinct` — `TreeSet<String>`, GmBuilder/
TestPlanBuilder'daki "ordinal-distinct" konvansiyonuyla birebir) → her biri için
`protected final {Entity}Repository {entity}Repository` alanı + ctor parametresi; ctor gövdesinde
atama. `public abstract Result<{Ret}> execute({Op}Command request)`. Javadoc'ta kanonik guard-çağrı
sırası referansı (idempotency→authz→validation→external-input→rule→entity+invariant→persist→emit→
return) — **Guards'ın kendisi ÇAĞRILMAZ** (task §7 out-of-scope; yalnız yorum).

### 5.3 Human seam — `humanHandlerJava`
`src/main/java/app/{module}/{op}/{Op}Handler.java` (writeIfAbsent): `{Op}HandlerBase`'i extend eder;
ctor `super(...)` ile repository'leri iletir; `execute` gövdesi birebir marker:
`throw new UnsupportedOperationException("{opId}: iş mantığı doldurulacak");`. Öncesinde
`writer.migrateSeamIfFlat(oldFlat, newSlice)` çağrılır (`oldFlat` = düz eski yol
`src/main/java/app/{module}/{Op}Handler.java`, brownfield güvence). `@Component` KONULMADI — kayıt
tamamen Wiring `@Bean`'inde (anti-pattern §8 uyumlu; testle doğrulandı).

### 5.4 Endpoint — `endpointJava`
`{Op}Endpoint.java` (WriteAlways) yalnız `visibility != internal` **VE** en az bir `serving.protocol
== "rest"` varsa emit edilir; aksi halde dosya YOK (WriteAuditLog: internal + rest-serving yok →
ikisi de tek başına yeterli sebep). Her rest serving girdisi için bir route metodu:
POST/PUT/PATCH → `@RequestBody {Req} request`; GET/DELETE → route `params` listesindeki adlar
`@PathVariable`, kalan `signature.params` `@RequestParam` (aynı sırada) — gövdede
`new {Req}(...)` ile paramlardan kurulur. Gövde her zaman tek satır:
`return ResultHttp.toHttp(handler.execute(request));`. `realized("serving", "{op}:rest")` yalnız
emisyon gerçekleştiğinde. Non-rest serving (ör. CreateInvoice'un `@grpc()` girdisi) sessizce
atlanır — **T4.5 kapsamı** (task notu; bu task'ta `Unsupported` çağrılmadı).

### 5.5 Wiring bean kayıtları
`{Module}Wiring` — op-slice döngüsü içinde biriktirilen `Set<String>` (ordinal-sıralı, dedup) import
+ `StringBuilder` bean-metot bloğu, döngü sonunda tek dosyaya yazılıyor:
- `@Bean {Op}Handler {op}Handler({Entity}Repository ...) { return new {Op}Handler(...); }` — her op
  için, görünürlükten bağımsız (internal dahil DI'a girer).
- `@Bean {Op}Endpoint {op}Endpoint({Op}Handler h) { return new {Op}Endpoint(h); }` — **yalnız**
  Endpoint dosyası emit edildiyse.
Repository parametre tipi, entity'nin kendi modülünde ise (fixture'daki tüm durumlar) aynı paket
olduğundan import GEREKMEZ; farklı modüldeyse cross-module import eklenir (genel durum için
uygulandı, fixture'da tetiklenmedi).

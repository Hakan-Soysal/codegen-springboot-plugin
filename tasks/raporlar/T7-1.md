### 1. Fixture harness — `gen/test-java/app/Fixture.java` (writeAlways, unconditional)

`SpringEmitter.fixtureJava(GenConfig)` — hafif Spring context bootstrap: statik iç
`Boot` sınıfı (`@Configuration @EnableAutoConfiguration @Import(GeneratedBootstrap.class)`) +
`SpringApplicationBuilder(Boot.class).web(WebApplicationType.NONE)`. `@SpringBootTest`
KULLANILMADI (anti-pattern §8). H2 override: `spring.datasource.url` property'si
`"jdbc:h2:mem:" + java.util.UUID.randomUUID()` (runtime-ifade; UUID emit-time ÜRETİLMEDİ —
golden bozulmaz). `testDbProvider` postgres/sqlserver ise `TEST_DB` env bağlantısı (whitelist-dışı
davranış .NET paritesi, task §5.1).

Yardımcılar:
- `<T> Result<T> run(Class<?> handlerClass, Object request)` — reflection ile `execute` metodunu
  bulur, bean'i context'ten çözer, invoke eder. `request==null` ise (ARRANGE seam henüz
  doldurulmadıysa) canonical-ctor + primitive-default değerlerle **default-instantiate** eder
  (CoreTemplate1 `Fixture.RunAsync`/`Instantiate` paritesi) — böylece iskelet, ARRANGE
  doldurulmadan da derlenir VE koşar (task §2 Why: doldurucu skill akışı bozulmasın).
- `<E> E get(Class<E> entity, String id)` — `EntityManagerFactory` bean'i + JPQL
  `SELECT e FROM {SimpleName} e`, ilk kayıt. `id` v1'de kullanılmaz (varlık-tabanı kontrolü;
  alan-düzeyi lookup task §7 out-of-scope).
- `reset()` / `close()`.

### 2. TestPlan→JUnit emisyonu — `emitTests(gm, writer, report)`

`gm.testPlan()`'ın üç listesi (processTests / orphanFlowTests / orphanOpTests) gezilir. Klasör
adlandırması (task §5.2 birebir): `process` (sabit) / `orphanflow_{id-lowercase}` /
`orphanop_{id-lowercase}`. Sınıf adı Pascal(`{id}`) + `Test`/`Arrange`.

Her test için (`emitOneTest`):
- Prerequisites içinde **Single-dışı** (Ambiguous/Missing) adım varsa: **iskelet/seam EMİT
  EDİLMEZ**; her offending prereq için `report.unsupported("test-prereq", "{id}: {entity}
  creator={ambiguous|missing}", ...)` (task §5.4'ün literal id-şablonu birebir).
- Aksi halde: prereq'ler (SINGLE creator'lar, topo-sıra TestPlanBuilder'dan gelir) + runSequence
  birleşik, 1-tabanlı index'li adım listesine (`ResolvedStep`) çözülür → owned iskelet
  (`gen/test-java/.../{Name}Test.java`, writeAlways) + ARRANGE seam
  (`src/test/java/.../{Name}Arrange.java`, writeIfAbsent).
- Owned iskelet 3-faz: ARRANGE-çağrı yorumu + prereq `fixture.run({CreatorOp}Handler.class,
  arrange.requestN())` çağrıları / ACT: runSequence `fixture.run({Op}Handler.class,
  arrange.requestN())` / ASSERT: **her** `run()` sonucu (`assertInstanceOf(Success.class, rN)`,
  prereq+act ikisi de) + WriteSet'teki her entity için `assertNotNull(fixture.get(Entity.class,
  null))`.
- ARRANGE seam: her adım için `public {Op}Command/{Op}Query requestN() { return null; // Arrange
  {op}: **doldurulacak** — ... }` — marker birebir korunur, **yalnız burada** (owned iskelette
  ASLA — anti-circularity, standing rule §6).
- Emit edilen testler **census'a girmez** — `report.realized("test", ...)` **ÇAĞRILMAZ** (task
  §5.2'nin açık kararı: "test construct'ı census'ta YOK").

## Test-emisyon kapsam kararları (spec iç-tutarsızlıkları — standing rule §4, çözülmedi, raporlanıyor)

1. **Task §5.2 ARRANGE-yorum metni vs. §5.3 method-adı.** §5.2 3-faz yorumu ARRANGE satırını
   `"{Name}Arrange.arrange{Op}() çağrıları"` diye tarif ediyor; ama §5.3 method adını kesin olarak
   `request{i}()` (index-tabanlı, op-adı DEĞİL) olarak tanımlıyor. İkisi birbiriyle örtüşmüyor.
   **Karar:** §5.3'ün kesin biçimi (`requestN()`) UYGULANDI; §5.2'nin yorum satırı
   **açıklayıcı/dokümantasyon metni** olarak alındı (literal method adı değil) — owned iskelette
   emit edilen yorum satırı `// ARRANGE (human): {ArrangeClass}.requestN() çağrıları — ...` şeklinde,
   §5.3 ile TUTARLI isimlendirmeyi kullanıyor.
2. **CoreTemplate1 `DotnetEmitter.cs:235`** fiilen `report.Realized("test", ...)` çağırıyor — ama
   task metni "parite: .NET de test'i census'a saymaz" diyor (referans kaynakla ÇELİŞİYOR).
   **Karar:** Task'ın açık talimatı (census'a GİRMEZ) izlendi — `report.realized("test",...)`
   ÇAĞRILMADI. Bu, task talimatının (referans-pattern'e göre) önceliklendirildiği bilinçli bir
   karardır.
3. **"Handler-üst-tipi" (task §5.1) mevcut değil.** Bu kod tabanında her op'un kendi
   `{Op}HandlerBase` soyut sınıfı var; TÜM handler'ların paylaştığı ortak bir arayüz/soyut tip YOK.
   **Karar:** `Fixture.run(Class<?> handlerClass, Object request)` — jenerik `Class<?>` +
   reflection'la `execute` metodu bulunuyor (CoreTemplate1'in kendi `RunAsync<THandler>`'ı da benzer
   şekilde ortak arayüz olmadan reflection kullanıyor — pattern paritesi korunuyor).
4. **"Her run sonucu Success" kapsamı.** Task metni ASSERT'i yalnız "her run" diye tarif ediyor,
   ACT'e (runSequence) mi yoksa TÜM `fixture.run(...)` çağrılarına (prereq dahil) mi uygulanacağı
   açık değil. **Karar:** TÜM `fixture.run(...)` çağrıları (prereq+act) için Success assert'i
   üretildi — daha eksiksiz/savunulabilir bir v1 test iskeleti (prereq'in de başarılı kurulduğunu
   doğrular).
5. **Unsupported id-formatı.** Task §5.4 literal şablon veriyor:
   `"{name}: {entity} creator={ambiguous|missing}"` (tekil entity) — .NET referansı ise
   `{scope}_{name}` + virgülle-birleştirilmiş offender listesi kullanıyor. **Karar:** Task'ın
   literal şablonu birebir uygulandı; birden fazla offending prereq varsa **her biri için ayrı bir
   unsupported entry** üretiliyor (aggregate DEĞİL) — daha granüler ve grep'lenebilir.
